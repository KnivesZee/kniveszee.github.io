BFPRT

Posted on 2018-11-19 | Post modified: 2018-12-07 | Visitors:
Words count in article: 2,511 | Reading time ≈ 10

在一大堆数中求其前k大或前k小的问题，简称TOP-K问题。而目前解决TOP-K问题最有效的算法即是BFPRT算法，其又称为中位数的中位数算法，该算法由Blum、Floyd、Pratt、Rivest、Tarjan提出，最坏时间复杂度为 $O(n)$ 。

1. 问题的引入

荷兰国旗问题：我们将乱序的红白蓝三色小球排成有序的红白蓝三色的同颜色在一起的小球组。这个问题之所以叫荷兰国旗问题，是因为我们可以将红蓝白三色小球想象成条状小物，有序排列后正好组成荷兰国旗。

其实它本身就是快排的那个Partion过程，就是将一个数，小于它的放左边，大于它的放右边，额外空间复杂度O(1)做到。

这个问题如何解决，首先想到的就是进行排序，然后再找，但这样做的话就来了一个 $O(nlogn)$ 的代价，而BFPRT算法可以以 $O(n)$ 来找到这个数。

解决问题前，先介绍几个相关的基础概念——Partition

2. Partition

这个是快排中的Partion，是在一个数组中随机选择一个数，将这个序列中小于等于这个数的放在左边，大于的放在右边，荷兰国旗问题中，可以将它改为小于的放左边，等于的放中间，大于的放右边，分成3块儿。

3. 一种优秀的解决方法

若将一个数分成3个区域了，便可以找到等于部分的边界，假如这个数组长1000, 等于部分是500~600，而k是550,这样便一次命中，返回结果，若k为200,则在 $[0,500]$ 间进行Partion,反之在 $[600,1000]$ 之间进行Partion递归求解

这种方式不是我们的BFPRT算法，但是它已经足够优秀了，其实它就是快排的一个改进。

为什么是上面那种方式进行寻找，还是以上面那个条件进行排，假设我们要找的是第200小的数，即k=200，那第200小的数肯定在这三块儿的最左块儿 $[0,500]$ 间，它不可能再在这右边的区域了

但是这种算法的时间复杂度完全取决于划分的有多优良，下面举个好的情况。

我们选择的划分值刚好在中间区域，那么左边和右边刚好是同等规模的，大概都是 $n/2$ 。

但是不可能每次都选择的这么好，可能存在左边很长，右边很短。最极端的是每次都打偏，每次都只排除一个数，那么就会变成 $O(n^2)$ 的算法。

所以这个方法，虽然是随机选择值进行Partion，但是它是随机的，概率的。将概率这个因素算上，所以它整体的时间复杂度是一个基于概率的表达式，长期期望已经能够做到 $O(n)$ 了，而不是 $O(nlogn)$ ，因为快排不管怎么样都要把左右两边都得排好序，但这种方法它只需要管一边，所以它和快排的时间复杂度不一样，在只选择一边的情况下，它的期望是 $O(n)$ 的。

如何算的时间复杂度？

若最坏情况： $T(n)=T(n-1)+O(n)$ ,它就会变为一个 $O(n^2)$
特别好的情况： $T(n)=T(n/2)+O(n)$ ,它就会变为一个 $O(n)$

所以随机选择一个数，既能避免最好最差，也可能能达到最好最差，完全随机的，而这个随机值打到每个位置上是等概率的，于是表达式是一个概率累加的复杂度，经过化简，数学上概率的长期期望是 $O(n)$ 的。

4. BFPRT的流程

上面那种概率的方式很好，但是有没有严格的不含概率的，用不着算长期期望的 $O(n)$ 算法？——有，它就是BFPRT算法，这个算法本身非常的精彩。

经典解法是随机选择一个数，再进行Partion。而BFPRT算法的差别就是选择划分值的策略不一样，之后的处理方式完全相同，那么BFPRT算法是如何选择这个划分的值呢？

下面是真个算法的步骤，定义 int bfprt(arr, k),其中arr是传入的数组

  1. 整个数组中，先逻辑分组，相邻的5个数分一组，[0,4], [5-9]一组，如此递推，最后单下来的数单独成为一组
  2. 每一个小组内排序，跨组之间不保证有序，大概差不多N/5个组，只是5个数之间排序，代价为O(1),所以这个过程总共的，在O(N)时间内可以拿下
  3. 把每5个数的中位数拿出来，构成一个新的数组new_arr，新数组长度为N/5
  4. 非常关键，递归调用BFPRT算法 num = bfprt(new_arr, len(new_arr)/2 )
  5. 就用上面那个num做划分，将小于num的放左边，等于的放中间，大于的放右边，然后看命中k了没有，命中了直接返回，没命中的话选一侧再走

TIPS：

  1. 为什么是分成5个，因为BFPRT算法的发明者正好是5个人，当然你也可以3个一分组，7个一分组，就这么任性
  2. 上面第四步是在干什么？就是我们想找到这中位数拿出来组成的N/5大小数组中的中位数。而我们BFPRT的算法功能是什么？在一个数组中找到第k小的数。如果数组长度为10，那么中位数便是第5小的数，如果数组长度为12，那么数组的中位数便是第6小的数，所以这步就是找到这些中位数数组中的中位数，当这个值拿到了
  3. 第三步是一个递归行为，可求得一个划分值

BFPRT EG:

原串32146 01235 234

  1. 分组：32146 01235 234
  2. 排序：12346 01234 234
  3. new_arr = [3,2,3]
  4. 调用BFPRT( [3,2,3], 3/2 ) 求下标为1，第二小的数得到 num = 3
  5. 将原数组按照小于3的放左边，等于的放中间，大于的放右边，看命中k了没有，命中了直接返回，没命中的话选一侧再走

5. BFPRT为什么要选择如此特殊的值进行划分

这个值到底好在哪里呢？

首先分析一下算法的复杂度：

  1. 第一步分组 $O(1)$
  2. 第二步组内排序 $O(n)$
  3. 组成新数组 $O(n)$ ，因为一共有n/5个中位数，组成一个新的数组，长度为n/5所以，它也是 $O(n)$ 的
  4. 这一步：若我们把BFPRT的数据规模作为n来估计的话，BFPRT本身用 $T(n)$ ,这一步就用的是 $T(n/5)$ ，假设这个数最后返回为P
  5. 这一步使用Partion划分: $O(n)$
  6. 命中返回，没命中走左/右。我们按照最差的情况来估计，左侧代表原数组中有多少个数要比P小，那左侧最坏的情况就是在数组中最多有多少个数比P要小啊，右侧是最多有多少个数比P要大。因此我们如果要估计最多有多少个数比P要小，那么我们可以估计至少有多少个数比P要大来达成目的。那么如何来估计至少有多少个数比P要大呢？根据图（画在纸里）想一下，将所有的点想为原数组，然后在这个框里面选择了中位数x，如果真个数组个数为N，那么框中的数有n/5个，所以在这个n/5个大小的数组中，有n/10个数比这个中位数x要大，每一个比x大的数，在自己的组内也是中位数，在组内又有两个数比这个数要大。所以最坏的情况下，整体n个数中至少有3n/10的数要比x要大，同理至多有7n/10比x要大。这样我们就确定了左部分和右部分最大的规模。此时，左右部分分别最多就是花费 $T(7n/10)$

于是 $T(n)=T(n/5)+T(7n/10)+O(n)$ ,最后算出来是 $O(n)$ 的，具体在算法导论中有讲这个东西，这里便不赘述。

6. BFPRT算法的应用：

最多的便是在一个数组中找到最大的/最小的数，最多的方法就是用堆做，但用堆做的话就是O(nlogn)或者O(nlogk)了，实际上这个题目的最优解便是BFPRT。
